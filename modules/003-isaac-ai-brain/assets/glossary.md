# Module 3 Glossary: Isaac-Specific Terms

## A

**Articulation**: A collection of rigid bodies connected by joints that can move relative to each other. In Isaac Sim, robots are typically represented as articulations.

**API (Application Programming Interface)**: Set of definitions and protocols for building and integrating application software, including Isaac Sim's Python API.

## B

**Behavior Tree**: A hierarchical tree structure that defines actions and conditions for robot behavior, often used in Isaac ROS for complex task execution.

## C

**CUDA**: Parallel computing platform and programming model developed by NVIDIA for general computing on GPUs, essential for Isaac Sim's rendering and Isaac ROS's perception pipelines.

**Compute Express Link (CXL)**: High-speed interconnect technology that can be used in Isaac-enabled systems for improved memory coherency.

## D

**Domain Randomization**: Technique of randomizing simulation parameters (lighting, textures, physics properties) to improve transfer from simulation to reality.

**Differential Drive**: A common robot drive mechanism where each side of the robot has its own wheel(s), controlled independently for steering.

**Docker**: Platform for developing, shipping, and running applications in containers, often used for Isaac ROS deployment.

## E

**Extension**: Modular component in Isaac Sim that adds functionality to the core application, such as new UI panels or simulation features.

## F

**Fastrak**: A real-time, GPU-accelerated physics simulation engine integrated into Isaac Sim.

**FPGA (Field-Programmable Gate Array)**: Reconfigurable hardware often used in Isaac-enabled robots for specialized computation.

## G

**GPU (Graphics Processing Unit)**: Specialized electronic circuit for manipulating computer graphics and image processing, crucial for Isaac Sim's rendering and Isaac ROS's perception processing.

**GXF (Gems Executive Framework)**: NVIDIA's framework for building perception and AI applications, used by Isaac ROS.

## H

**Humble Hawksbill**: ROS 2 distribution that Isaac ROS supports, providing the middleware for robot communication.

**Humanoid Robot**: A robot with a human-like body structure, including legs, torso, arms, and head.

## I

**Isaac Sim**: NVIDIA's robotics simulation application built on NVIDIA Omniverse for developing, testing, and validating AI-based robotics applications.

**Isaac ROS**: Collection of packages and tools that accelerate perception and navigation workloads for ROS 2 using NVIDIA GPUs.

**Isaac ROS NITROS**: NVIDIA Isaac Transport for ROS, optimizing data transport between nodes using type adaptation.

**IMU (Inertial Measurement Unit)**: Sensor that measures specific force, angular rate, and sometimes magnetic field, used for robot localization.

## L

**LiDAR (Light Detection and Ranging)**: Sensor technology that measures distance by illuminating a target with laser light and measuring the reflection.

**Lattice Planner**: Path planning algorithm that uses pre-computed motion primitives for efficient navigation.

## M

**Micro-ROS**: Lightweight version of ROS 2 for microcontrollers, sometimes integrated with Isaac systems.

**Middleware**: Software that provides common services and capabilities to applications, such as ROS 2's DDS implementation.

## N

**Nav2 (Navigation 2)**: ROS 2 package for robot path planning and navigation, often integrated with Isaac ROS perception.

**NVIDIA Jetson**: AI computing platform designed for autonomous machines, commonly used with Isaac ROS.

**NVIDIA RTX**: Graphics processing technology that enables real-time ray tracing and AI-enhanced rendering in Isaac Sim.

## O

**Omniverse**: NVIDIA's simulation and collaboration platform that Isaac Sim is built upon.

**Occupancy Grid**: 2D or 3D grid-based representation of space where each cell indicates the probability of occupancy.

**Odometry**: Method for estimating change in position over time, typically from wheel encoders or visual features.

## P

**Perception Pipeline**: Series of processing steps that transform sensor data into meaningful information about the environment.

**PID Controller (Proportional-Integral-Derivative)**: Control loop mechanism used for robot actuator control.

**Point Cloud**: Set of data points in 3D space, typically generated by LiDAR or depth sensors.

**PhysX**: NVIDIA's real-time physics engine used in Isaac Sim for realistic physics simulation.

## R

**ROS (Robot Operating System)**: Flexible framework for writing robot software that provides hardware abstraction and device drivers.

**ROS 2**: The second generation of ROS with improved security, real-time support, and multi-robot capabilities.

**RViz**: 3D visualization tool for ROS that can display robot models, sensor data, and other information.

## S

**SLAM (Simultaneous Localization and Mapping)**: Computational problem of constructing or updating a map of an unknown environment while simultaneously keeping track of an agent's location.

**Stereolabs ZED**: Stereo camera often used with Isaac ROS for depth perception.

**Simulation-to-Reality Transfer**: Process of applying knowledge or models trained in simulation to real-world robotic systems.

**USD (Universal Scene Description)**: 3D scene description and file format used by Isaac Sim and Omniverse.

## T

**Tegra**: NVIDIA's system-on-chip (SoC) platform used in robotics applications with Isaac software.

**TF (Transforms)**: ROS package that keeps track of coordinate frames over time, essential for robot perception and navigation.

**Triton Inference Server**: NVIDIA's inference server that can be integrated with Isaac ROS for AI model deployment.

## V

**VSLAM (Visual Simultaneous Localization and Mapping)**: SLAM approach that uses visual sensors (cameras) for localization and mapping.

**VDB (Volume Database)**: Sparse volumetric data structure often used for 3D mapping in Isaac applications.

**Viewport**: The display area in Isaac Sim where the 3D scene is rendered.

## W

**Waypoint**: A location in space that a robot is commanded to navigate to.

**Wheeled Inverted Pendulum (WIP)**: Model for balancing robots like segways, sometimes simulated in Isaac Sim.
